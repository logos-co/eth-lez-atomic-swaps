# Upstream Issues for lez-framework

Issues to file on https://github.com/jimmy-claw/nssa-framework.
Discovered while porting the LEZ HTLC program to use the framework.

---

## Issue 1: Generated `main()` is not cfg-gated, preventing inline unit tests

**Component**: `lez-framework-macros`

### Problem

The `#[lez_program]` macro generates a `fn main()` unconditionally (line ~229 of `lez-framework-macros/src/lib.rs`). When `cargo test` runs on the guest crate, this `main()` becomes the binary's entry point and immediately calls `read_nssa_inputs()`, which invokes risc0 guest syscalls that don't exist on the host. The test process aborts with:

```
panicked at risc0-zkvm-platform/.../syscall.rs: not implemented
thread caused non-unwinding panic. aborting.
```

This prevents any `#[cfg(test)] mod tests { ... }` in the same binary crate.

### Current workaround

Users must:
1. Wrap `risc0_zkvm::guest::entry!(main)` and `#[lez_program]` in `#[cfg(not(test))]`
2. Extract all domain logic into standalone `_impl` functions outside the module
3. Test the `_impl` functions instead of the actual handlers

This means signer/init validation generated by the macro is **never tested** on the host.

### Suggested fix

In `expand_lez_program`, wrap the generated `main_fn` in `#[cfg(not(test))]`:

```rust
let expanded = quote! {
    #enum_def
    pub const PROGRAM_IDL_JSON: &str = #idl_json;
    mod #mod_name {
        use super::*;
        #(#other_items)*
        #(#handler_fns)*
        #(#validation_fns)*
    }
    #idl_fn
    #[cfg(not(test))]
    #main_fn
};
```

And make the handler functions + validation functions `pub` so tests can call them directly:

```rust
// In tests:
let result = lez_htlc::lock(maker, escrow, hashlock, taker_id, amount);
// This exercises the real handler including macro-generated signer/init checks
```

### Impact

Every program using `#[lez_program]` is affected. Without this, inline unit tests are impossible and the macro-generated validation logic goes untested.

## Issue 2: Multi-seed PDA combination uses XOR instead of SHA-256

**Component**: `lez-cli/src/pda.rs`

### Problem

Multi-seed PDAs are combined by XOR-folding all resolved 32-byte seeds:

```rust
fn xor_bytes(a: &[u8; 32], b: &[u8; 32]) -> [u8; 32] {
    let mut result = [0u8; 32];
    for i in 0..32 {
        result[i] = a[i] ^ b[i];
    }
    result
}

let combined = resolved
    .iter()
    .skip(1)
    .fold(resolved[0], |acc, seed| xor_bytes(&acc, seed));
```

XOR has two properties that make it unsuitable for PDA derivation:

1. **Commutativity**: `XOR(A, B) == XOR(B, A)` — seed order doesn't matter, so `pda = [literal("vault"), account("user")]` produces the same PDA as `pda = [account("user"), literal("vault")]`. Programs that rely on seed ordering for namespace separation will collide.

2. **Self-cancellation**: `XOR(X, X) == 0` — if two seeds happen to resolve to the same 32 bytes, they cancel out entirely, collapsing to `PdaSeed::new([0; 32])`.

### Suggested fix

Replace XOR with `SHA-256(seed1 || seed2 || ...)`:

```rust
use sha2::{Sha256, Digest};

let mut hasher = Sha256::new();
for seed in &resolved {
    hasher.update(seed);
}
let combined: [u8; 32] = hasher.finalize().into();
let pda_seed = PdaSeed::new(combined);
```

This is order-dependent, collision-resistant, and consistent with how Solana and other systems derive multi-seed PDAs.

### Impact

Any program using multi-seed PDAs is affected. The multisig program (`lez-multisig`) uses multi-seed PDAs today. This is a **breaking change** for existing deployed PDAs — would need a migration path or version flag.

## Issue 3: `#[account(pda = ...)]` does not generate guest-side PDA verification

**Component**: `lez-framework-macros`

### Problem

The `#[account(pda = arg("hashlock"))]` attribute currently only flows into the IDL metadata. The `generate_validation` function (line ~649) only generates signer and init checks — **no PDA derivation or account ID comparison**.

This means the guest program blindly trusts that the client supplied the correct PDA account. A malicious or buggy client could pass any account ID for a PDA-annotated parameter, and the guest would process it without rejection.

### Current workaround

Programs must manually verify PDA derivation in handler logic:

```rust
let expected_pda = AccountId::from((&program_id, &PdaSeed::new(hashlock)));
assert!(escrow.account_id == expected_pda, "escrow PDA mismatch");
```

But this defeats the purpose of the declarative `pda = ...` annotation.

### Suggested fix

In `generate_validation`, for each account with `pda_seeds`, emit:

```rust
{
    let __expected_pda = {
        // Resolve seeds to [u8; 32] values
        // Combine using SHA-256 (per issue #N)
        // Derive AccountId from (program_id, PdaSeed)
    };
    if accounts[#idx].account_id != __expected_pda {
        return Err(LezError::PdaMismatch {
            expected: __expected_pda,
            actual: accounts[#idx].account_id,
        });
    }
}
```

This requires:
- The program ID to be available in the guest (can be read from the NSSA environment or passed as a const)
- A new `LezError::PdaMismatch` variant
- Seed resolution logic in guest codegen (for `arg(...)` seeds, use the deserialized instruction fields; for `literal(...)`, pad at compile time)

### Impact

Without this, the `pda = ...` annotation is purely cosmetic — it documents intent but doesn't enforce it. This is a security gap for any program that relies on PDA accounts being correctly derived.

## Issue 4: `AccountId` not recognized as a known IDL type

**Component**: `lez-framework-macros`

### Problem

In `rust_type_to_idl_json` (line ~764), `AccountId` is not in the explicit match list alongside `u8`, `u128`, `String`, `ProgramId`, etc. It falls through to the generic catch-all:

```rust
other => format!("{{\"defined\":\"{}\"}}", other),
```

So an instruction argument of type `AccountId` becomes `{"defined":"AccountId"}` in the IDL, rather than being recognized as a native 32-byte identifier. This means:

- `lez-cli` can't parse `AccountId` arguments from the command line
- `lez-client-gen` may generate incorrect client code for `AccountId` fields

### Suggested fix

Add `AccountId` to the known type list, mapping to `[u8; 32]` or a dedicated `"account_id"` IDL type:

```rust
match ident.as_str() {
    "u8" | "u16" | "u32" | "u64" | "u128" | ... => { ... }
    "ProgramId" => "\"program_id\"".to_string(),
    "AccountId" => "\"account_id\"".to_string(),  // NEW
    other => format!("{{\"defined\":\"{}\"}}", other),
}
```

And in `lez-cli/src/parse.rs`, handle `"account_id"` by parsing a hex string or base58 into `AccountId`.

### Impact

Any program that takes `AccountId` as an instruction argument (e.g., `taker_id` in an HTLC Lock instruction) is affected. The IDL is generated correctly in terms of structure, but downstream tools can't interpret the type.
